## A-1 Ultmarcの沿革と業界背景

### 日本アルトマーク（Ultmarc）の歴史

Ultmarc（日本アルトマーク）は1962年設立以降、製薬企業向けマーケティングリサーチを基盤に、1972年からDCF（Doctor Computer File）を構築してきた歴史を持ちます。

| 時期 | 出来事 |
|------|--------|
| 1962年 | 日本アルトマーク設立。製薬企業向けマーケティングリサーチを開始 |
| 1972年 | 全国医師医療施設コンピューターファイル（DCF）作成開始 |
| 2013年 | NTTドコモが約26億円で約77.5%の株式取得（医療ICT事業強化目的） |
| 2015年 | NTTドコモによる完全子会社化 |
| 2019年 | エムスリー（m3.com）がNTTドコモから全株式を66億円で取得。m3.comの医師会員27万人とMDBを統合し、製薬業界向けマーケティング支援を強化 |

現在はエムスリーグループの完全子会社として運営されており、MDB（医療データベース）は業界標準データとして継続提供されています。

### MDB（医療データベース）の概要

| 項目 | 内容 |
|------|------|
| 会員数 | 195社以上の製薬・医療機器企業 |
| 運営モデル | 企業からの情報提供をUltmarcが一元管理・更新し、共通コード（DCFコード等）で共有する業界共同メンテナンスモデル |
| カバー範囲 | 病院約8,000件、診療所約106,000件、福祉施設約65,000件 |
| 料金体系 | 有料サブスクリプション制（MDB会員制）。基本料金＋抽出件数に応じた従量課金（例：1件120円程度） |
| Veeva経由利用 | VeevaライセンスにUltmarcデータ利用料が含まれる形で別途課金 |

### MDBファイルの詳細

**DCF（Doctor Computer File）**
- 全国の医療施設（病院・診療所）と所属医師の基本情報を収録
- 施設データ：施設名・住所・診療科・病床数・DPCフラグ・がん診療連携拠点病院フラグ等
- 医師データ：氏名・役職（教授・講師等）・所属部科・勤務先施設コード

**DSF（Drug Store File）**
- 全国の薬局・薬店の基本情報を収録
- 店舗名・住所・代表者・チェーン本部・調剤薬局フラグ等

**PCF（Pharmacist Computer File）**
- 全国の病院内薬剤師の基本情報を収録
- 氏名・役職・所属病院・資格情報等

### 「OpenData」という名称について

Veeva OpenDataの「Open」は、データが第三者と共有しやすく、オープンなアクセス・パートナープログラムを意味します。従来の複雑な三者契約を避けられる点が特徴です。「無料」を意味するわけではなく、**レコード数（HCP/HCO件数）と対象国数に基づくサブスクリプション制の有料サービス** です。

## A-2 VeevaとSalesforce Life Sciences Cloudの競争比較

### 背景

Veeva CRMは元来Salesforceプラットフォーム上で構築されていましたが、現在は自社基盤の **Vault CRM** への移行を進めています（**2030年までのプラットフォーム分離期限**）。一方、Salesforceは2024年に **Life Sciences Cloud** を本格投入し、Veevaの領域に直接参入しています。

### アーキテクチャ思想の比較

| 観点 | Veeva | Salesforce Life Sciences Cloud |
|------|-------|-------------------------------|
| 設計思想 | 業界特化の閉じたエコシステム | 多業界基盤のライフサイエンス拡張 |
| 強み | Vault-CRM一体統制、GxPネイティブ、即戦力 | AI（Agentforce）先行、Data Cloud、パートナー規模 |
| データ統合 | Vault間のネイティブ連携 | Data Cloud + Informatica（80億ドル買収）による柔軟統合 |
| AI | Veeva AI / Link（展開中） | Agentforce Life Sciences（HCPスケジューリング等） |
| コンテンツ統制 | PromoMats-CLMネイティブ連携 | パートナーエコシステムで構築 |
| 主な採用実績 | トップ20製薬の9社（Vault CRM） | Novartis、AstraZeneca等 |

### Salesforceの勝ち筋

1. **AI/Agentforceの先行**：HCPスケジューリング・コールプランニング・コンテンツ提案・音声ログをAIエージェントで自動化
2. **データ統合の優位**：Data Cloud + Informaticaで、IQVIA・EMR・患者データを一元化
3. **パートナーエコシステム**：athenahealth・H1・Infinitus.aiと連携。AppExchangeの数百アプリが強み
4. **移行支援とコスト**：Veeva-on-Salesforceユーザーの2030年分離期限を活かし、低コスト移行＋AIアップグレードを提案

### Veevaの防衛力と弱点

| Veevaの強み | Veevaの弱点（リスク） |
|---|---|
| 業界特化のコンプライアンス・サンプリング・MSL機能 | AI競争の遅れ（Agentforceが先行） |
| 閉じたライフサイエンスエコシステムの深さ | 2030年分離期限での中小企業離脱リスク |
| Vault-CRM一体統制（GxPネイティブ） | エコシステムの狭さ（IQVIA/EMR/患者データ連携がSalesforceに劣る） |
| トップ20製薬9社の実績 | 高額ライセンス・カスタム制限が中小企業に不向き |

**競争の本質**：「業界特化の深さ」vs「汎用プラットフォームの広さと速さ」。

### データエンジニアへの示唆

Veevaは「閉じた統合の深さ」、Salesforceは「オープンな統合の広さ」をそれぞれ武器にしています。

- **Veeva環境での設計**：Vault API / VQLを中心とした統一インターフェースで設計できる
- **Salesforce環境での設計**：パートナー製品ごとのAPI仕様を個別に把握する必要がある
- **移行・共存シナリオ**：HCP/HCO識別子マッピングと、コンテンツ配布状態の整合性維持が最大の課題

## A-3 Veeva依存度の整理

製薬企業がVeevaに依存している機能と、他社で代替可能な機能を整理します。システム選定やリスク評価の参考として活用してください。

### 代替可能性の分類

| 機能・コンポーネント | Veeva依存度 | 代替可能性 | 備考 |
|---|---|---|---|
| Core CRM（活動記録・HCP/HCO管理） | 中 | 可能 | Salesforce・他社CRMで代替可能だが、移行コストが高い |
| Vault CRM（Vault-CRM統合） | 高 | 困難 | PromoMats-CLM連携はVeeva固有。代替時は別途コンテンツ統制基盤が必要 |
| Veeva Network / OpenData | 高 | 部分的 | 日本ではUltmarc直接契約で代替可能だが、グローバル統合は難しい |
| CLM（コンテンツ提示・ログ） | 高 | 困難 | Vault PromoMatsとの連携が前提。他社CLMではGxP保証に課題 |
| Approved Email | 高 | 可能 | MLRレビュー済みテンプレートの配信。別途承認フロー設計が必要 |
| Vault QualityDocs | 中〜高 | 可能 | MasterControl・Documentumなどの代替QMS製品が存在する |
| Vault Submissions / RIM | 高 | 困難 | eCTD申請特化製品（Extedo等）で代替可能だが、統合コストが高い |
| Vault Safety | 中〜高 | 可能 | Oracle Argus・ARISg等の代替PVシステムが存在する |
| Veeva AI / Link | 中 | 可能 | 展開中のため、他社AI（Salesforce Agentforce等）で代替する事例が増えている |
| Veeva Nitro（日本：Jitsushoka） | 中 | 可能 | Ultmarcデータ自体はAPI等で直接取得可能 |

### 日本市場における独自リスク

- **HCP/HCOデータ**：Veeva OpenDataが非対応のため、UltmarcとのVeeva連携に依存。Veeva離脱時はUltmarc直接連携への移行が必要。
- **Jitsushoka / Nitro**：Ultmarcデータとの日本固有のコネクタはVeeva経由が最も整備されており、代替時に追加開発が発生しやすい。
- **HCP Access**：Veeva CRM全体のエコシステムデータであるため、Veeva離脱時に業界横断的なアクセス指標が取得不可能になる。

### 総括

- **CRM単体**：他社CRMでも代替可能だが、コンテンツ統制・GxP保証の再設計が必要
- **Vault + CRM一体**：「文書承認→現場配信→活動記録→監査証跡」のエンドツーエンドをVeeva以外で実現するには、複数製品の統合設計が必要
- **日本のHCP/HCOデータ**：Veevaが直接グローバル提供するというより、ローカル提携（Ultmarc）や追加サービスで補う構造であり、データ側の依存はVeevaとUltmarc双方に分散している
