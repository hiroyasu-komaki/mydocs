3-1 Veeva Platform全体像 3-1 Veeva Platform Overview

Veevaは大きく Vault（コンテンツ・プロセス管理） と CRM（顧客接点管理） の2系統に分かれます。両者はマスタデータ（Veeva Network）を介して接続され、主に上市後フェーズで交差します。 Veeva is broadly divided into Vault (content & process management) and CRM (customer engagement). Both systems are connected via Veeva Network (master data) and primarily intersect in the post-launch phase.

3-2 データ責任分界（System of Record） 3-2 Data Ownership (System of Record)

設計原則：「Vault は regulated content の system of record、CRM は field engagement の system of action」。複数システムに同じデータが存在する場合、どちらが正であるかを業務ルールとして定義しないと、バージョン不整合・更新漏れが発生します。 Design principle: "Vault is the system of record for regulated content; CRM is the system of action for field engagement." When the same data exists in multiple systems, failing to define which is authoritative causes version conflicts and update gaps.

3-6 CRM系コンポーネントのデータフロー 3-6 CRM Component Data Flow

Veeva CRMは消費財CRMとは根本的に異なり、規制遵守下での情報提供・活動管理が目的。最大の違いは「GxP承認済みコンテンツのみ配信可能」であり、サンプリング規制・州法がシステムで強制されます。 Veeva CRM is fundamentally different from consumer-goods CRMs — its purpose is regulated information provision and activity management. The key distinction: only GxP-approved content can be delivered; sampling rules and state laws are system-enforced.

ライフサイクル設計の考え方Lifecycle Design Principles

Draft In Review Approved Expired
ワークフロー設計では「誰がどのステートでどのアクションを実行できるか（ロールと権限）」を業務フローと対応させる必要があります。 Workflow design must map "who can perform which action in which state" (roles and permissions) to the actual business process.

CRM連携における重要ポイントKey point for CRM integration

PromoMatsでApprovedになった瞬間にCLMへ同期、Expiredになった瞬間にCLMから撤回。単純なデータコピーではなく、状態遷移に応じた制御付き同期として設計が必要です。Approval triggers CLM sync; Expiry triggers CLM withdrawal. This is not a simple data copy — design as state-change-controlled conditional sync.

変更管理との連動Change Control Integration

承認後の文書変更（CMC変更・ラベリング変更）は規制当局への届け出要否判断と連動して設計します。Active Dossierが「現在有効なドキュメントセット」、Submissionsが「変更申請コンテンツ」という役割分担を崩さないことが重要です。 Post-approval document changes (CMC or labeling) must be designed in coordination with regulatory filing requirements. It is essential to maintain the role separation: Active Dossier = "currently valid document set"; Submissions = "change application content."
