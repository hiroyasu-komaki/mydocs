# 連邦型ITガバナンス体系 - 要点まとめ

## 🎯 コアコンセプト

**連邦型ガバナンス**は、中央集権型と分散型の中間に位置し、**グローバルな一貫性**と**ドメインの機敏性**を両立させるガバナンスモデルです。

<br>

## 📊 ガバナンス体系の全体構造

```
Level 0: ガバナンス憲章（1文書）
└─ Level 1: エンタープライズポリシー（6文書）
   └─ Level 2: ガードレール（44文書）
```

### Level 1: 6つのコアポリシー
1. **POL-100**: セキュリティ・コンプライアンス（10ガードレール）
2. **POL-200**: ITアーキテクチャ（8ガードレール）
3. **POL-300**: データガバナンス（8ガードレール）
4. **POL-400**: ITサービスマネジメント（7ガードレール）
5. **POL-500**: IT投資・ポートフォリオ管理（4ガードレール）
6. **POL-600**: リスク・コンプライアンス管理（6ガードレール）

<br>

## 🚦 ガードレールの3ゾーン設計

各ガードレールは自律判断の境界を明確化します：

| ゾーン | 意味 | 特徴 |
|-------|------|------|
| 🟢 **グリーン** | ドメインが自由に判断可能 | 事後報告のみ、迅速な実行 |
| 🟡 **イエロー** | 中央への相談・承認が必要 | 明確な期限、承認者、必要資料 |
| 🔴 **レッド** | 禁止または即時エスカレーション | 技術的強制、違反時措置 |
---

<br>

#### 実例（IT投資判断）
- 🟢 **100万円未満、既存予算内** → ドメインで即決（四半期報告のみ）
- 🟡 **100万〜1,000万円** → CTO承認（2週間以内）
- 🔴 **1,000万円以上** → IT投資委員会（月次開催）

<br>

## 🔄 移行の難易度ランキング

現状から連邦型への移行パターン別：

| 順位 | パターン | 難易度 | 推奨度 |
|------|---------|--------|--------|
| 1位 🥇 | 分散型 → 連邦型 | 最も容易 | ◎ 強く推奨 |
| 2位 🥈 | 連邦型 → 分散型 | 容易 | △ 条件付き |
| 3位 🥉 | 連邦型 → 中央集権型 | 中程度 | ○ 状況次第 |
| 4位 | 中央集権型 → 連邦型 | やや困難 | ○ 日本企業で多いパターン |
| 5位 | 分散型 → 中央集権型 | 困難 | △ 慎重に |
| 6位 🚫 | 中央集権型 → 分散型 | 最も困難 | ✕ 非推奨 |

**注**: 製薬業界など規制産業では、段階的アプローチが不可欠

### 移行を検討すべきシグナル（中央集権型→連邦型の場合）

以下の兆候が見られる場合、連邦型への移行を検討すべきです：

#### ビジネス影響
- ✅ 意思決定の遅さが市場機会の損失を招いている
- ✅ 地域市場のニーズへの対応が競合に遅れている
- ✅ イノベーションや実験的取り組みが抑制されている

#### 組織課題
- ✅ 時差により重要な判断が数日遅延している
- ✅ 限られた中央IT人材が全社のボトルネックになっている
- ✅ 海外拠点から自律性を求める声が強まっている

#### 戦略的要因
- ✅ 国際展開の加速が経営戦略の中核になっている
- ✅ 地域ごとの規制・コンプライアンス要件が複雑化している
- ✅ M&Aで獲得した組織の統合が課題になっている

## 📐 ガードレール設計の4ステップ

### フェーズ1: 要件定義
1. **ポリシー要件の分解**
   - 上位ポリシーから統制したい事項を抽出
   - リスクと守りたい価値を明確化

2. **リスク軸の特定**
   - 主要なリスク次元を列挙（金額、影響範囲、技術新規性、複雑性、期間など）
   - 最も重要な2-3個に絞り込む

### フェーズ2: ゾーン設計
3. **グリーンゾーン定義**（最初）
   - 「完全に安全」な条件を定義
   - 必要なアクション、判断者、判断期限を明記

4. **レッドゾーン定義**（2番目）
   - 「絶対に譲れない」線を引く
   - 禁止事項、技術的強制、違反時の措置を定義

5. **イエローゾーン定義**（最後）
   - グリーンとレッドの間を階層化（通常2-3層）
   - 各階層の条件、承認者、承認期限を設定

### フェーズ3: 検証
6. **例外プロセスの設計**
   - 例外を認める条件（緊急事態、想定外の状況など）
   - 手順、承認者、期限を明確化

7. **境界の検証**
   - 典型的なシナリオでテスト（5-10ケース）
   - グレーゾーン（判断が曖昧なケース）の排除

### フェーズ4: 導入準備
8. **ステークホルダー検証**
   - ドメインリーダーへのレビュー依頼
   - フィードバック収集と調整

9. **最終調整と文書化**
   - 数値の根拠を記録
   - 完全性・整合性・実行可能性をチェック

## Key of Success

1. **明確な境界線**
   - グレーゾーンを作らない
   - 誰が見ても判断が一致する基準

2. **測定可能な基準**
   - 数値・条件で明確化
   - 主観的な表現を避ける

3. **段階的移行**
   - 特に日本企業では文化変革に配慮
   - クリティカルな領域から順次実施

4. **継続的改善**
   - 四半期レビューで調整
   - 実運用からのフィードバックを反映

5. **ドメインの信頼**
   - 自律性を尊重しつつ秩序を提供
   - 「権限を奪う」のではなく「支援を提供する」姿勢
